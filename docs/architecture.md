# Architecture

> This document explains the moving parts and why they're arranged the way
> they are. The README has the high-level view; this one drills into
> each component, the trust boundaries, and the data flows.
>
> **Scope.** Everything below describes the **PoC topology** (single-node
> RKE2 in a Vagrant + VirtualBox VM, local-path storage, in-cluster MinIO).
> For the production topology — Terraform against a real hypervisor/cloud,
> cloud-init + Ansible bootstrap, 3-node control plane behind an external
> LB, Longhorn on dedicated worker disks, off-cluster MinIO — see
> [`ha-pattern.md`](ha-pattern.md) and
> [`production-gaps.md`](production-gaps.md).

## Component map

```
HOST (your machine)
└── VirtualBox VM (openSUSE Leap 15.6, 6 vCPU / 10 GB)
    └── RKE2 single-node Kubernetes (v1.35.x)
        │
        ├── INGRESS plane
        │   ├── MetalLB                ─ L2 advertisement, IP pool 192.168.56.20-29
        │   └── Traefik                ─ Ingress controller, TLS via cert-manager
        │
        ├── PKI plane
        │   ├── cert-manager           ─ controllers + webhook + cainjector
        │   ├── selfSigned-bootstrap   ─ Issuer that signs the lab CA cert
        │   ├── lab-ca-certificate     ─ self-signed root CA (90d lifetime, renew 30d)
        │   └── lab-ca-issuer          ─ ClusterIssuer that signs every workload cert
        │
        ├── STORAGE plane
        │   └── local-path-provisioner ─ default StorageClass (rancher.io/local-path)
        │                                creates PVs as host-path dirs under /var/lib/rancher/k3s/storage
        │
        ├── GITOPS plane
        │   └── ArgoCD                 ─ application-controller, repo-server, server, redis, dex
        │       ├── root Application   ─ the ONLY resource Terraform applies after the helm-release
        │       └── ~17 child Apps     ─ discovered from argocd/apps/*.yaml; sync-wave ordered
        │
        ├── SECRETS plane
        │   ├── Vault (HA Raft, 1 replica)  ─ KV v2 engine at path 'secret/', Kubernetes auth method
        │   └── External Secrets Operator   ─ controller + webhook + cert-controller
        │                                     ClusterSecretStore 'vault' wraps the Vault SA token
        │
        ├── DATA plane
        │   ├── MinIO                  ─ single instance, 10 GB local-path PVC
        │   │                            buckets: cnpg-backups, vault-backups
        │   │                            scoped users: cnpg-bk-*, vault-sn-*
        │   └── CloudNativePG operator ─ manages Postgres Clusters (CRD)
        │       └── keycloak-db Cluster ─ primary + sync replica, barmanObjectStore to MinIO
        │
        ├── APPLICATION plane
        │   ├── Keycloak operator      ─ vendored upstream manifests, watches keycloak ns
        │   └── Keycloak app           ─ StatefulSet rendered by the operator, 1 instance
        │
        └── OBSERVABILITY plane
            ├── Prometheus             ─ 24h retention, 5 GB PVC, cluster-level scrapes only
            ├── Grafana                ─ admin password from Vault via ESO; Loki + Prometheus datasources
            ├── kube-state-metrics
            ├── node-exporter (DaemonSet)
            ├── Loki SingleBinary      ─ 72h retention, 5 GB PVC, filesystem storage
            └── Promtail (DaemonSet)   ─ tails /var/log/pods on the host, pushes to Loki
```

## Trust boundaries

| Boundary | Crossing point | Auth | Transport |
|---|---|---|---|
| Host browser → cluster | Traefik on 192.168.56.20:443 | per-app (Keycloak login, ArgoCD password, etc.) | HTTPS, lab-ca-issuer-signed cert |
| Cluster pod → kube-apiserver | ServiceAccount tokens | RBAC | HTTPS, cluster CA |
| ESO → Vault | K8s ServiceAccount via Vault `kubernetes` auth method | Vault policy `eso-readonly` (read-only on secret/data/*) | HTTPS in-cluster (Vault dev disables TLS; production would use server-side TLS) |
| Init script → Vault | Vault root token (one-time, generated by `vault operator init`) | Root | HTTPS port-forward |
| CNPG → MinIO | AWS-style key/secret per cluster (`cnpg-bk-*`) bound to `cnpg-backup-policy` | MinIO IAM | HTTP in-cluster |
| Vault snapshot → MinIO | AWS-style key/secret (`vault-sn-*`) bound to `vault-snapshot-policy` | MinIO IAM | HTTP in-cluster |
| Keycloak app → Postgres | Postgres user + password from `keycloak-db-app-user` Secret (ESO-managed) | PG auth | TCP in-cluster, NetworkPolicy-restricted |
| Prometheus → kubelet | Prometheus SA token | RBAC | HTTPS |

## Data flows

### Bootstrap (cold start)

The reviewer runs ONE of two parallel entry points. Both end at the same
Healthy cluster — the only difference is which IaC tool drives steps 1-5.

```
# Path A — Terraform (default)
PS> powershell -ExecutionPolicy Bypass -File .\scripts\ps\bootstrap.ps1 -WaitForHealthy
sh> ./scripts/bash/bootstrap.sh --wait-for-healthy

# Path B — Pulumi+Go mirror (see docs/pulumi-mirror.md)
PS> $env:PULUMI_CONFIG_PASSPHRASE = 'lab-passphrase-change-me'
PS> powershell -ExecutionPolicy Bypass -File .\scripts\ps\bootstrap-pulumi.ps1 -WaitForHealthy
sh> ./scripts/bash/bootstrap-pulumi.sh --wait-for-healthy
```

What the wrapper does, step by step:

```
1. Pre-flight  → checks vagrant, terraform|pulumi+go, kubectl, helm on PATH

2. IaC apply   → Path A:  terraform init && terraform apply -auto-approve
                 Path B:  pulumi login --local && pulumi up --yes --stack dev
                 Both:
                   - Vagrant up (build VM, install RKE2)
                   - kubeconfig exported to ./kubeconfig
                   - Helm install of ArgoCD into argocd ns
                   - kubectl apply -f argocd/root-app.yaml
                   - root Application discovers argocd/apps/*.yaml
                   - child Applications start syncing in wave order

3. Wait        → poll vault-0 pod until phase=Running (up to 20 min)

4. init-vault  → Path A:  scripts/ps/init-vault.ps1   (or .sh)
                 Path B:  same script, called by the Pulumi bootstrap wrapper
                 Steps inside init-vault:
                   → port-forward vault-0:8200
                   → vault operator init -key-shares=1 -key-threshold=1
                   → save root token + unseal key to terraform/vault-keys.json (gitignored)
                   → vault operator unseal
                   → vault kv put secret/<seed paths>
                   → vault auth enable kubernetes
                   → vault write auth/kubernetes/config token_reviewer_jwt=<eso SA token>
                   → vault write auth/kubernetes/role/external-secrets
                   → vault policy write eso-secret-reader
                   → vault write auth/kubernetes/role/vault-snapshot
                   → vault policy write vault-snapshot-policy

5. ESO catches up (cluster-side, automatic)
   → ClusterSecretStore 'vault' transitions Valid
   → every ExternalSecret in every namespace resolves
   → K8s Secrets materialize (minio-root-credentials, keycloak-db-app-user,
     keycloak-bootstrap-admin, vault-snapshot-minio-creds, etc.)
   → workload pods that were Pending on missing secrets un-pend and reach Ready

6. (Optional with -WaitForHealthy) wait for every Argo Application Healthy
   → up to 25 min; Keycloak is typically last due to Quarkus build phase
```

Re-running either bootstrap script is **idempotent**: `terraform apply` /
`pulumi up` skip what already exists, `init-vault.ps1` detects existing
init/unseal/role and skips them. Useful after an interrupted run.

### Steady state — request flow for a Keycloak login

```
browser
  → https://keycloak.lab.test                                    (TLS handshake with lab-ca cert)
  → Traefik (192.168.56.20:443)
  → Traefik routes by Host: header → keycloak-service:8080
  → Keycloak StatefulSet pod (HTTP, X-Forwarded-* trusted)
  → Postgres connection pool → keycloak-db-rw:5432
  → primary Postgres pod (CNPG-managed) on local-path PVC
```

### Steady state — backup flow

```
CNPG instance manager (in keycloak ns)
  → wal-archive every WAL segment as it rotates
  → daily 02:00: `cnpg backup` triggered by ScheduledBackup CR
  → POST to MinIO via S3 API
  → cnpg-bk-* user authenticates (key from minio-cnpg-backup-creds Secret)
  → MinIO writes to bucket cnpg-backups/ under the cnpg-backup-policy ACL

vault-snapshot CronJob (in vault ns, daily 03:00)
  → initContainer (hashicorp/vault):
      login via k8s auth (SA: vault-snapshot)
      vault operator raft snapshot save → /shared/vault-snapshot.snap
  → container (mc):
      mc cp /shared/vault-snapshot.snap → local/vault-backups/<timestamp>.snap
      mc rm --older-than 7d  (retention)
  → vault-sn-* user authenticates (key from vault-snapshot-minio-creds Secret)
  → MinIO writes to bucket vault-backups/ under the vault-snapshot-policy ACL
```

### Steady state — observability flow (currently disabled)

```
node-exporter (DaemonSet, host network)
  → /metrics on :9100
  ←─── Prometheus scrapes every 30s (intra-monitoring ns)

every pod's stdout/stderr
  → kubelet writes to /var/log/pods/<ns>_<pod>_<uid>/<container>/0.log on host
  → Promtail (DaemonSet) tails the file via hostPath mount
  → POST to http://kps-loki:3100/loki/api/v1/push

Grafana
  → datasource 'Prometheus' (built into kps)         → PromQL queries
  → datasource 'Loki' (registered via ConfigMap)     → LogQL queries
```

## Sync wave ordering

ArgoCD applies child Applications in wave order. Earlier waves must reach `Healthy` before later ones start (in theory — Argo treats this as a hint, not a strict barrier; in practice the wait-and-retry config makes the ordering robust).

| Wave | Applications | Why this wave |
|---|---|---|
| **-4** | `local-path` | StorageClass must exist before any PVC binds |
| **-3** | `cert-manager`, `metallb` | Controllers + CRDs must register before Issuers/Pools |
| **-2** | `cert-manager-issuers`, `traefik`, `metallb-config` | Need CRDs from wave -3 |
| **-1** | `vault`, `external-secrets`, `cnpg-operator`, `keycloak-operator` | All operators install in parallel |
| **0**  | `external-secrets-config`, `argocd-ingress` | ClusterSecretStore needs ESO CRDs; ingress needs Traefik |
| **1**  | `minio` | Needs ESO config from wave 0 (root creds materialize before MinIO pod starts) |
| **2**  | `keycloak` | Needs MinIO (CNPG backup target) + ESO (DB creds + admin) + operators |
| **3**  | `vault-backup` | Needs Vault unsealed (wave -1 + init-vault) AND MinIO bucket+user (wave 1) AND ESO (wave 0) before the snapshot CronJob can authenticate end-to-end |
| **4**  | `networkpolicies` | After every namespace exists — applying default-deny to a namespace before its workload exists would block legitimate traffic during pod startup |
| **5**  | `monitoring` | After everything else is up — Prom + Grafana have nothing to monitor otherwise |

## What I deliberately did NOT add

- **An ingress controller other than Traefik.** RKE2 ships Traefik by default but we disable it (in `/etc/rancher/rke2/config.yaml`) and install our own via ArgoCD so the version + config are pinned in Git.
- **A service mesh.** mTLS between pods is a common production add-on but adds significant complexity (sidecar injection, certificate rotation, traffic policies). NetworkPolicies cover the lab's L4 security needs.
- **A shared/multi-tenant Postgres database for multiple components.** Each workload uses the storage backend its own operator supports natively:
  - **Keycloak** has a dedicated **CNPG Cluster** (the only thing that actually needs a relational DB)
  - **ArgoCD** stores its declarative state in **Kubernetes ConfigMaps + Secrets** (it doesn't use SQLite); operational state lives in **Redis**
  - **Vault** uses **Raft** as its storage backend (built into Vault, no external DB)

  This avoids the "spin up a generic shared Postgres and have N workloads contend for it" anti-pattern. Each workload's storage matches what its operator natively expects, so upgrades, backups, and operations stay isolated and operator-managed.
- **An authenticated / private Git repository.** The lab's repo is **public on GitHub** and ArgoCD pulls anonymously over HTTPS — fine for a lab. Production GitOps would use a **private repository** with one of:
  - Git provider OAuth token (GitHub App, GitLab deploy token) materialized as a K8s Secret in the `argocd` namespace and referenced by ArgoCD's `repo-credentials` template
  - SSH deploy key (also via K8s Secret + `repo-credentials`)
  - **The credential itself stored in Vault and materialized by ESO** — same pattern this lab uses for every other secret, so Git credentials aren't plaintext in any cluster manifest

  An air-gapped cluster (no GitHub egress) additionally needs a **cluster-internal Git server** (Gitea, Forgejo, or GitLab CE) that ArgoCD's repo-server can reach.
- **Longhorn (or any replicated block storage).** Resources limitations on a single-node lab. See `production-gaps.md`.

## Things that look like complexity but aren't

- **Wrapper Helm charts** (`charts/keycloak/`, `charts/minio/`, `charts/monitoring/`). These wrap upstream charts as a dependency plus add a few of our own templates (ExternalSecrets, Ingresses, NetworkPolicies, Namespace labels). They're standard Helm dependency pattern — `helm dependency build` vendors the upstream .tgz into the wrapper, then the wrapper renders both its templates and the subchart's.
- **`unsupported.podTemplate` on the Keycloak CR.** Keycloak operator doesn't expose podTemplate fields directly; the `unsupported` block is the official escape hatch for pod-level customizations (security context, additional volume mounts, etc.). It's "unsupported" only in the sense that the operator team doesn't promise backward compatibility on it — the feature itself is documented and stable.
- **Sync wave 4 for NetworkPolicies.** Three reasons:
  1. **Target namespaces must exist first.** `manifests/networkpolicies/policies.yaml` writes policies in `vault`, `traefik`, `argocd` etc. — those namespaces are created in earlier waves (-4 to 2). Wave 4 guarantees they're all there.
  2. **Apply order = debugging order.** Cluster works without policies first → then apply policies → observe what specifically breaks.
  3. **Smallest allow-set, written from observed traffic.** By wave 4 the actual traffic graph is fully exercised (every controller has done its initial cluster scan, every webhook has been called, every operator has talked to its CRs). Writing policies at this point means we describe the real graph, not a guessed one.

## Where things live

- **Code that creates the platform**: `terraform/` / `pulumi/` + `argocd/`
- **Code that configures the platform**: `manifests/` (flat YAML) + `charts/` (Helm wrappers)
- **Code that operates the platform**: `scripts/`
- **Code that explains the platform**: `docs/` + `README.md`
